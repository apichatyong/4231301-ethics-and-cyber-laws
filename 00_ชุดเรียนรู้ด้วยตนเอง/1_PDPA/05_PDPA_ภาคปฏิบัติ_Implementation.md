## 1. หลุมพรางนิยามที่อาจารย์เน้น (เข้าใจก่อนลงมือ)

**ทำไมต้องแยกข้อมูลส่วนบุคคล vs ข้อมูลทั่วไปให้ออก?**
เพราะถ้าแยกไม่ออก องค์กรจะ "ทำเกิน" (เปลืองทรัพยากรไปคุ้มครองข้อมูลที่ไม่ต้องคุ้มครอง) หรือ "ทำขาด" (ละเลยข้อมูลที่กฎหมายคุ้มครอง → โดนปรับ)

⭐ **คำถามดักในห้อง: "อีเมลเป็นข้อมูลส่วนบุคคลไหม?"**
คำตอบคือ **"แล้วแต่ว่าระบุตัวบุคคลได้หรือไม่"**
- `recruitment-th@coffee-concepts.co.th` (อีเมลกลางของแผนก) → **ข้อมูลทั่วไป** เพราะชี้ไปที่ "ตำแหน่งงาน" ไม่ใช่ตัวคน
- `kittisak.thamrugsasit@coffee-concepts.co.th` (มีชื่อ-สกุล) → **ข้อมูลส่วนบุคคล** เพราะระบุตัวคนได้

**ข้อมูลทั่วไป (ไม่อยู่ใต้ PDPA):** เลขทะเบียนบริษัท · เบอร์/อีเมล/ที่อยู่สำนักงาน · ข้อมูลนิรนาม (Anonymous) · ข้อมูลผู้ตาย · ยอดขายรวม · จำนวนผู้เข้าชมเว็บแบบรวม (ไม่ระบุใคร)

## 2. ⭐ 5 ขั้นตอนทำองค์กรให้ PDPA Compliant (พร้อมคำอธิบาย "ทำไม/ทำอย่างไร")

### ขั้น 1 — ตั้งคณะทำงาน
**ทำไม:** PDPA มีศัพท์เฉพาะเยอะ การอบรมทั้งบริษัทพร้อมกันทำได้ยาก จึงต้องตั้ง "ทีมแกน" ก่อน
**ใครบ้าง:** DPO + ฝ่ายกฎหมาย + IT/ดิจิทัล + ผู้บริหารที่ออกนโยบายได้ + **Champion**
> 💡 **Champion** = ตัวแทนจากแต่ละฝ่ายที่ "ใช้ข้อมูลจริง" (HR, การตลาด, ขาย, ลูกค้าสัมพันธ์, E-Commerce) ทำหน้าที่ให้ข้อมูลกิจกรรมของฝ่ายตน + ประสานงาน — เป็นกลไกสำคัญที่ทำให้สำรวจข้อมูลได้ครบ

### ขั้น 2 — อบรมพนักงาน
ให้ทีมเข้าใจหลักการ + แยกแยะ DC/DP/DS + รู้จักข้อยกเว้น ก่อนลงมือสำรวจ

### ขั้น 3 — สำรวจข้อมูล (PDPA Fact-Finding Questionnaire)
ส่งแบบสำรวจให้ **Champion แต่ละฝ่าย** กรอกว่าฝ่ายตนเก็บ/ใช้/เปิดเผยข้อมูลอะไรบ้าง บันทึกในรูป **ROPA ของฝ่าย** แล้วนำมารวมเป็น **ROPA ของบริษัท**
> เป้าหมาย: ให้ DPO เห็น "ภาพรวมการประมวลผลข้อมูลทั้งองค์กร"

### ขั้น 4 — Data Mapping & Gap Analysis
เอาข้อมูลจากแบบสำรวจมาทำแผนผังการไหลของข้อมูล (Data Flow) แล้ว **หาช่องว่าง** ว่าจุดไหนยังไม่ตามกฎหมาย
> ตัวอย่างจริง: Coffee Concepts จัดทำ *"List of Key Issues and Recommendations"* = ผลของ Gap Analysis

### ขั้น 5 — จัดทำเอกสาร/เครื่องมือปิดช่องว่าง
→ ดูชุดเอกสารข้อ 3

## 3. ⭐ ชุดเอกสารสำคัญ (อธิบายว่าแต่ละตัวคืออะไร + ใช้ตอนไหน)

| เอกสาร | คืออะไร / ใช้ตอนไหน |
|---|---|
| **Privacy Notice** | **ประกาศ**ให้เจ้าของข้อมูลรู้ว่าเราเก็บอะไร-ทำไม-นานเท่าไหร่-ส่งให้ใคร-มีสิทธิอะไร (ตาม ม.23) ต้องเข้าถึงง่าย ภาษาชัด → เช่น หน้า Privacy Notice ของ **Starbucks** |
| **Retention Policy** | นโยบายกำหนด "เก็บนานเท่าไหร่แล้วลบ" เพื่อไม่ให้เก็บเกินจำเป็น |
| **Consent Form** | แบบขอความยินยอม ใช้เฉพาะกิจกรรมที่ต้องอาศัยฐานยินยอม เช่น **การตลาด / วิเคราะห์พฤติกรรม / เก็บข้อมูลอ่อนไหว** — ต้องแยกช่องออกจากเงื่อนไขบริการอื่น |
| **แบบคำร้องขอใช้สิทธิ (DSAR Form)** | ฟอร์มให้เจ้าของข้อมูลยื่นใช้สิทธิ ม.30–36 (มีช่องระบุว่ายื่นเองหรือเป็นตัวแทน + แนบสำเนาบัตร ปชช./หนังสือมอบอำนาจ) |
| **ROPA** | บันทึกรายการประมวลผลทั้งองค์กร (ม.39) — เช่นไฟล์ Excel ของ Coffee Concepts |
| **แผนรับมือเหตุละเมิด** | ขั้นตอนเมื่อข้อมูลรั่ว → แจ้ง สคส. ใน 72 ชม. + เยียวยา (ม.37) |
| **DPA / Data Sharing Agreement** | สัญญาตอนส่งข้อมูลออกนอกองค์กร (ดูข้อ 4) |

## 4. ⭐⭐ DPA vs Data Sharing Agreement (จุดที่อาจารย์เน้นมาก — ออกสอบ)

ทั้งคู่ใช้ตอน "ส่งข้อมูลส่วนบุคคลออกนอกองค์กร" แต่ต่างกันที่ **ใครเป็นเจ้าของอำนาจตัดสินใจ**

| | **DPA (Data Processing Agreement)** | **Data Sharing Agreement** |
|---|---|---|
| ใช้เมื่อ | ส่งให้ **outsource/ผู้รับฝากข้อมูล** ที่ทำ**ตามคำสั่งเรา** | ทั้งสองฝ่าย **ใช้/แลกเปลี่ยนข้อมูลร่วมกัน** ต่างคนต่างตัดสินใจ |
| สถานะ | เรา = **DC**, อีกฝ่าย = **DP** | **ทั้งคู่เป็น DC** |
| เทียบกับ | คล้าย NDA แต่เน้นคุ้มครองข้อมูลส่วนบุคคล | — |
| เนื้อหาขั้นต่ำ | วัตถุประสงค์ส่งต่อ, ประเภทข้อมูล, เงื่อนไขประมวลผล, ระยะเวลาเก็บ, มาตรฐานความปลอดภัย, **ความรับผิดหากข้อมูลรั่ว** | คล้ายกัน + กำหนดความรับผิดร่วม |

**ตัวอย่างแยกให้เห็นชัด:**
- บริษัทส่งข้อมูลพนักงานให้บริษัท **Payroll จ่ายเงินเดือน** → Payroll ทำตามคำสั่ง = **DP** → ใช้ **DPA**
- บริษัทส่งข้อมูลลูกค้าให้ **บริษัทวิจัยตลาด** เอาไปวิเคราะห์เอง → ถ้าวิจัยตัดสินใจเองว่าจะวิเคราะห์อย่างไร = อาจเป็น DC ร่วม → **Data Sharing Agreement**
> 💡 ควรฝัง DPA ไว้ในกระบวนการ **จัดซื้อจัดจ้าง** ทุกครั้งที่ต้องส่งข้อมูลออกนอกองค์กร

## 5. 📌 กรณีศึกษาจริงที่อาจารย์ยก (อธิบายละเอียด)

### 🎯 กรณีศึกษา 1 — ระบบสะสมคะแนนสมาชิกห้างสรรพสินค้า
**สถานการณ์:** ร้านกาแฟ (Coffee Concepts) ตั้งอยู่ในห้าง ลูกค้าสมัครสมาชิก/สะสมคะแนนผ่านระบบของ **ห้าง** โดยใช้เบอร์โทร/บัตรสมาชิก

**กระบวนการ 4 ขั้น:** ① สมัคร/สะสมคะแนน → ② จัดเก็บข้อมูล (เก็บในระบบของ**ห้าง**) → ③ ใช้คะแนน (เสียบบัตร/สแกนที่เครื่อง POS) → ④ คุ้มครองข้อมูล

**ประเด็นวินิจฉัย (สำคัญ):**
- **ห้างสรรพสินค้า = ผู้ควบคุมข้อมูล (DC)** เพราะเป็นเจ้าของระบบและตัดสินใจ
- **ร้านกาแฟ = เป็นเพียง "ผู้ช่วยอำนวยความสะดวก" (Facilitator)** — แจ้งลูกค้า/แนะนำการสมัคร/ช่วยใช้สิทธิผ่านบัตรเท่านั้น
- **ร้านค้าไม่ได้เก็บข้อมูลสมาชิกไว้เอง** และไม่มีสิทธิเข้าถึงฐานข้อมูลของห้าง
- ✅ **ความเสี่ยง PDPA ของร้านค้าจึงอยู่ระดับต่ำ** เพราะ: ไม่บันทึกเบอร์/รูปลูกค้าไว้เอง · ไม่นำข้อมูลออกนอกระบบ · ไม่มีสิทธิเข้าถึงฐานข้อมูลสมาชิก

> ⚠️ **ข้อควรระวังที่อาจารย์เตือน:** หากร้านค้าไปปฏิบัติผิด เช่น **แอบจดเบอร์ลูกค้าเอง** หรือใช้ POS เชื่อมต่อจัดการข้อมูลสมาชิกของห้าง → สถานะทางกฎหมายอาจเปลี่ยน (กลายเป็นผู้เก็บ/ประมวลผลข้อมูล) ต้องตรวจสอบกระบวนการทำงานจริงประกอบ
>
> **บทเรียน:** สถานะ DC/DP **ไม่ได้ดูที่ "ป้ายชื่อ" แต่ดูที่ "ใครตัดสินใจ/ใครเก็บข้อมูลจริง"**

### 🎯 กรณีศึกษา 2 — กล้องวงจรปิด (CCTV) กับ PDPA
**สถานการณ์:** ร้านติดกล้องวงจรปิด → ภาพใบหน้าลูกค้า = **"ข้อมูลส่วนบุคคล"** ต้องปฏิบัติตาม PDPA

**5 หลักสำคัญในการติด CCTV ให้ถูกกฎหมาย:**
1. **มีวัตถุประสงค์ชัดเจน** (เช่น เพื่อความปลอดภัย/ป้องกันอาชญากรรม)
2. **แจ้งเจ้าของข้อมูล** — ติดป้าย **"CCTV in Operation / บริเวณนี้มีการบันทึกภาพ"**
3. **เก็บเท่าที่จำเป็น** — โดยทั่วไป **30 วัน**
4. **รักษาความปลอดภัยของภาพ** — จำกัดผู้เข้าถึง
5. **ไม่ใช้ข้อมูลเกินวัตถุประสงค์**

- **ฐานการประมวลผล:** ส่วนใหญ่ใช้ **ฐานประโยชน์โดยชอบด้วยกฎหมาย (ม.24(5))** ไม่ต้องขอความยินยอม แต่ต้องแจ้ง (ป้าย) + ทำหน้าที่ DC ครบ
- **DC/DP:** เจ้าของสถานที่ = DC; บริษัทรับติดตั้ง/ดูแลระบบกล้อง = DP
- **Do's:** ติดป้ายแจ้ง, ตั้งมุมกล้องเท่าที่จำเป็น, จำกัดสิทธิเข้าถึง · **Don'ts:** ส่อง CCTV เข้าพื้นที่ส่วนตัวคนอื่น, เผยแพร่ภาพมั่ว
- **ตัวอย่างป้าย CCTV จริง (Coffee Concepts Retail):** ระบุผู้ควบคุมข้อมูล + วัตถุประสงค์ + **ระยะเวลาเก็บ 30 วัน** + QR code ลิงก์ไป Privacy Notice

> ⚠️ การติดกล้องในบ้านส่วนตัวเพื่อดูเองอาจเข้า**ข้อยกเว้น ม.4** (ใช้ส่วนตน/ครอบครัว) → ไม่อยู่ใต้ PDPA แต่ถ้าส่องไปที่สาธารณะ/บ้านคนอื่นต้องระวัง

### 🎯 กรณีศึกษา 3 — เอกสารจริงในการใช้สิทธิ & ส่งต่อข้อมูล
Workshop ยกตัวอย่างฟอร์มจริง:
- **แบบคำร้องขอใช้สิทธิ (DSAR Form)** ของ Coffee Concepts — มีช่องระบุว่ายื่นเอง/เป็นตัวแทน + แนบสำเนาบัตร ปชช. / หนังสือมอบอำนาจ → สะท้อนหน้าที่รองรับสิทธิเจ้าของข้อมูล (ม.30–36)
- **Data Processing Agreement (DPA)** ตัวอย่าง — สัญญาเมื่อส่งข้อมูลให้ outsource

## 6. ❓ แนวข้อสอบ Workshop (10 คะแนน) พร้อมเฉลย

**ปรนัย/จับคู่:**
1. ร้านกาแฟในห้างเป็นเพียงผู้แนะนำการสมัครสมาชิก ไม่เก็บข้อมูลเอง → ร้านมีสถานะ/ความเสี่ยงอย่างไร? → *(เป็นเพียง Facilitator, ความเสี่ยง PDPA ต่ำ; ห้าง=DC)*
2. ภาพจากกล้องวงจรปิดเป็นข้อมูลส่วนบุคคลหรือไม่? → *(ใช่ ต้องปฏิบัติตาม PDPA)*
3. CCTV เก็บภาพได้นานเท่าใดโดยทั่วไป? → *(ประมาณ 30 วัน — เท่าที่จำเป็น)*
4. ส่งข้อมูลพนักงานให้บริษัท Payroll → Payroll เป็น? → *(ผู้ประมวลผล DP → ใช้ DPA)*
5. เอกสารที่ประกาศวัตถุประสงค์การเก็บข้อมูลให้เจ้าของข้อมูลทราบ คือ? → *(Privacy Notice)*

**อัตนัย (เคสประยุกต์ — โครง 4 ขั้น):**
> *"บริษัท A จ้างบริษัท B ทำการตลาดออนไลน์ โดยส่งรายชื่อ+เบอร์ลูกค้าให้ B ยิงโฆษณา จงระบุสถานะของ A/B, เอกสารที่ต้องทำ และฐานการประมวลผล"*
- **A = DC, B = DP** → ต้องทำ **DPA**
- การตลาด = ใช้ฐาน **ความยินยอม (ม.19)** → ต้องมี **Consent Form** + **Privacy Notice** ที่ลูกค้ายินยอมไว้แต่แรก
- ถ้า A ไม่เคยขอยินยอมเพื่อการตลาด = ส่งให้ B ไม่ได้ (ละเมิด)
- **จริยธรรม:** เคารพการกำหนดเจตนาตนเอง + โปร่งใส + ไม่ใช้ข้อมูลผิดวัตถุประสงค์

## 7. 🎯 มิติจริยธรรมภาคปฏิบัติ
PDPA Compliance ไม่ใช่แค่ "ทำเอกสารให้ครบเพื่อกันโดนปรับ" แต่คือการ **สร้างวัฒนธรรมเคารพข้อมูลของผู้อื่นทั้งองค์กร** — ตั้งแต่ผู้บริหารถึงพนักงานหน้าร้าน
เคสร้านกาแฟ/CCTV สอนว่า: **เก็บเท่าที่จำเป็น + แจ้งให้ชัด + ไม่เข้าถึงเกินบทบาท** = ลดความเสี่ยงและสร้าง **ความไว้วางใจ (Trust)** ให้ลูกค้า ซึ่งคือหัวใจที่แท้จริงของ PDPA
